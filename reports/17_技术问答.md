# DRIVE 项目技术问答

## 一、算法原理与架构设计

### 1.1 关于 Hybrid Training：SFT Loss 和 GRPO Advantage 的混合权重

**实际情况：本项目没有使用 SFT + GRPO 的混合目标函数。**

从训练脚本来看，Stage 1 使用的是纯 GRPO 训练（`--advantage_estimator group_norm`），没有启用 PTX（Pretrain Loss）辅助损失。虽然 OpenRLHF 框架提供了 `--ptx_coef`（默认 0.05）参数来混合 SFT Loss，但我们的训练脚本中没有指定 pretrain_data，因此 PTX 分支不会生效。

训练的目标函数实际上是：

$$L = L_{\text{GRPO}} + \beta \cdot D_{\text{KL}}(\pi_\theta \| \pi_{\text{ref}})$$

其中 KL 散度项起到了类似 SFT 正则化的作用——它约束策略不要偏离原始预训练模型太远，间接保留了预训练阶段学到的代码能力。

**如果引入 SFT 混合训练，权重分配的影响：**

- SFT 权重过高（如 ptx_coef > 0.5）：模型倾向于模仿参考解法的风格，收敛快但容易过拟合到训练数据的特定模式。v4 的表现就是一个例子——3 个 epoch 的重复训练导致训练集上 exec +69%，但测试集上 -18.7%。SFT 本质上是行为克隆，不鼓励探索。
- SFT 权重过低（如 ptx_coef < 0.01）：模型完全依赖 GRPO 的 reward 信号来学习，探索空间大但收敛慢。如果 reward 信号本身有噪声（如我们发现的 32% 误判率），模型可能学到错误的模式。
- 合理的做法：在训练初期用较高的 SFT 权重（0.1-0.2）稳定训练方向，随训练进行逐步衰减到 0.01-0.05，让 GRPO 信号主导后期优化。

**如何实现 SFT 权重的动态衰减：**

从代码层面分析，OpenRLHF 的 `ppo_actor.py:311` 中 loss 计算为：

```python
loss = actor_loss + kl_loss * kl_ctl
```

只有 GRPO policy loss + KL 正则化，没有 SFT 项。`--ptx_coef` 参数虽然在 `train_ppo_ray.py:381` 中定义了（默认 0.05），但在 GRPO 训练路径中没有被实际使用——因为 GRPO 不走 GAE 分支，也没有 pretrain_data 的加载逻辑。

要实现"SFT 权重从高到低衰减"，有三种可行方案：

**方案 A：修改 OpenRLHF 源码，加入 PTX Loss（正统做法）**

这是 InstructGPT 论文中的 PPO-ptx 方法。在 `ppo_actor.py` 的 `training_step` 中加入 SFT 项：

```python
# 原始 (line 311)
loss = actor_loss + kl_loss * kl_ctl

# 改为
ptx_loss = compute_nll_loss(model, sft_batch)  # 在参考解法上的 NLL loss
ptx_coef = max(0.01, 0.2 * (1 - global_step / total_steps))  # 线性衰减 0.2 → 0.01
loss = actor_loss + kl_loss * kl_ctl + ptx_loss * ptx_coef
```

需要额外准备 SFT 数据（APPS 参考解法），并在 experience maker 中同时采样 RL 数据和 SFT 数据。工作量大，侵入性强。

**方案 B：用 KL 系数衰减模拟（轻量近似）**

KL 散度项本质上约束策略不要偏离预训练模型。高 KL 系数 ≈ 高 SFT 权重（强制接近原始分布），低 KL 系数 ≈ 低 SFT 权重（允许自由探索）。只需修改 KL controller 为 step-aware 的衰减调度：

```python
# 在 training_step 中替换固定 kl_ctl
kl_ctl = max(0.005, init_kl_coef * (1 - step / total_steps))
```

改动极小（1 行），但只是近似——KL 约束的是"不要偏离太远"，SFT loss 是"要像参考解法一样"，方向不完全一致。

**方案 C：两阶段训练模拟衰减（零代码改动）**

不改框架，通过两次训练实现：
1. Phase 1（前 50 步）：`init_kl_coef=0.05`，强约束 ≈ 高 SFT 权重
2. Phase 2（后 100 步）：从 Phase 1 checkpoint 继续，`init_kl_coef=0.01`，弱约束 ≈ 低 SFT 权重

这是最实用的方案，不需要改任何代码，只需要跑两次训练脚本。

### 1.2 为什么选择 GRPO 而不是 PPO

**（1）计算资源：省去 Critic 网络**

这是最直接的原因。PPO 需要一个与 Actor 同等规模的 Critic（Value）网络来估计状态价值函数 $V(s)$，用于计算 GAE advantage。对于 3B 模型：

- PPO 需要：Actor (3B) + Ref (3B) + Critic (3B) + Reward Model = 至少 12B 参数在 GPU 上
- GRPO 需要：Actor (3B) + Ref (3B) + 远程 Reward Service = 6B 参数在 GPU 上

从训练脚本可以看到，我们显式设置了 `--critic_num_nodes 0 --critic_num_gpus_per_node 0`，完全不分配 Critic 资源。在 4×RTX A6000（48GB×4）的硬件条件下，省去 Critic 的 3B 参数意味着每张卡多出约 6-8GB 显存，这些空间可以用于更大的 rollout batch 或更长的 max_new_tokens。

**（2）采样效率：组内对比替代价值估计**

PPO 的 advantage 估计依赖 Critic 网络的预测质量。Critic 本身需要训练，在训练初期 Critic 不准确时，advantage 估计噪声很大，导致 Actor 更新方向不稳定。

GRPO 的 advantage 直接来自同一 prompt 的多个采样的 reward 对比：

$$A_i = \frac{r_i - \mu_{\text{group}}}{\sigma_{\text{group}} + \epsilon}$$

这个估计不依赖任何学习的网络，只要 reward 函数本身是准确的，advantage 信号就是准确的。从代码实现（experience_maker.py:266-267）可以看到，group_norm 就是简单的组内标准化。

**（3）代码生成的"高离散空间"特性**

代码生成的 reward 分布极度稀疏和离散——一个程序要么通过测试用例（1.0），要么不通过（0.0），几乎没有中间状态。这对 Critic 网络是灾难性的：

- Critic 需要学习一个平滑的价值函数 $V(s)$，但代码生成的 reward landscape 是高度不连续的（改一个字符可能从全对变成全错）
- Critic 的预测误差会直接传导到 advantage 估计，在离散 reward 下这个误差被放大
- GRPO 绕过了这个问题：它不需要预测"这个状态有多好"，只需要比较"这 8 个采样中哪个更好"

我们的实验数据也支持这一点：v5 训练中 group_reward_std 从 0.174 逐步降到 0.160，说明组内对比信号在持续提供有效的梯度方向，而不需要 Critic 的辅助。

### 1.3 大 Rollout（n=64）为什么在难题场景下更有效

从消融实验数据看，r64（90 题×64 样本）的 score 提升幅度是 r16（360 题×16 样本）的 2.2 倍（+0.0188 vs +0.0084）。

**统计学解释：**

GRPO 的 advantage 估计本质上是一个组内均值和标准差的估计问题。对于一个 prompt，如果真实的 reward 分布是 $R \sim \mathcal{D}(\mu, \sigma^2)$，那么：

- 组内均值的标准误差 = $\sigma / \sqrt{n}$
- 组内标准差的估计误差 ∝ $1 / \sqrt{2(n-1)}$

当 $n=16$ 时，均值标准误差 = $\sigma/4$；当 $n=64$ 时，均值标准误差 = $\sigma/8$。advantage 估计的精度提升了一倍。

**在难题场景下这个效应被放大：**

Competition 题目的 reward 分布极度稀疏——绝大多数采样的 exec_score = 0，只有极少数能部分通过。假设一个 competition prompt 的真实通过率是 5%：

- n=16 时：期望只有 0.8 个正样本。大概率 16 个采样全是 0 分，组内方差为 0，advantage 全为 0，**这一步完全没有梯度信号**。即使偶尔有 1 个正样本，advantage 估计也极不稳定（单点估计）。
- n=64 时：期望有 3.2 个正样本。有足够的正负样本对比，组内方差非零的概率大幅提升，advantage 信号更可靠。

从训练日志验证：r16 的 group_reward_std 均值 0.179，r64 的均值 0.185。虽然差异不大，但 r64 的 reward 趋势更健康（P3 均值 +0.0033 vs r16 的 +0.0021），说明 64 个样本确实能更稳定地捕捉到稀疏的正向信号。

**本质上，这是一个 bias-variance tradeoff：**

- r16（广度优先）：每步看 8 道题（128/16），每题 16 个样本。覆盖面广但每题的 advantage 估计方差大。
- r64（深度优先）：每步只看 2 道题（128/64），每题 64 个样本。覆盖面窄但每题的 advantage 估计更精确。

对于难题，"精确估计少数题目的梯度方向"比"粗糙估计大量题目的梯度方向"更有价值，因为粗糙估计在稀疏 reward 下很可能是纯噪声。

## 二、工程实现与分布式训练

### 2.1 Reward Service 的并发处理与沙箱机制

**架构设计：**

Reward Service 是一个独立的 Flask HTTP 服务（`reward_service.py`），监听 `0.0.0.0:5000`，通过 `--remote_rm_url http://localhost:5000/scores` 与 OpenRLHF 训练框架对接。

**并发处理：**

当前实现是单进程同步处理——Flask 默认使用单线程 WSGI 服务器，每次处理一个请求。OpenRLHF 的 experience maker 在 rollout 阶段会批量发送请求，但 `reward_service.py` 的 `/scores` 端点每次只处理单条样本（从代码 `score = result[0]` 可以看出）。

这在当前规模下不是瓶颈，因为：
- 代码执行（subprocess）本身是 I/O bound，单次执行 2 秒超时
- rollout 阶段是串行的（先生成所有 response，再逐个评分）
- 如果需要扩展，可以用 gunicorn 多 worker 部署，或改为批量评分接口

**沙箱机制（防止死循环和资源耗尽）：**

`reward_verifier.py` 的 `execute_code` 函数实现了三层防护：

1. **超时控制**：`subprocess.run(..., timeout=2.0)` — 任何代码执行超过 2 秒自动 kill。`while True` 或 O(n!) 暴力搜索都会被截断，返回 0.0 分。

2. **内存限制**：通过 `preexec_fn=_set_limits` 在子进程启动前设置 `RLIMIT_AS = 256MB`。如果代码试图分配超过 256MB 内存（如 `[0] * 10**9`），OS 会直接 kill 子进程。

3. **进程隔离**：每次执行都创建独立的 `subprocess`，代码写入临时文件执行。子进程崩溃不会影响 Flask 主进程。`finally` 块确保临时文件被清理。

**对训练流水线的保护：**

即使某个代码样本触发了死循环，最坏情况是该请求阻塞 2 秒后超时返回 0.0 分。Flask 主进程不受影响，下一个请求正常处理。训练流水线的 experience maker 会等待所有 reward 返回后才进入 training 阶段，所以单个超时最多延迟 2 秒，不会挂起整个训练。

**已知限制：**

- 没有网络隔离：恶意代码理论上可以发起网络请求（但训练数据是 APPS 公开数据集，风险极低）
- 没有文件系统隔离：代码可以读写 `/tmp` 等目录（但 256MB 内存限制和 2 秒超时限制了破坏范围）
- 如果要更严格的沙箱，可以用 Docker 容器或 nsjail，但会增加每次执行的启动开销

### 2.2 colocate_actor_ref 的显存优化

**`--colocate_actor_ref` 的作用：**

在 OpenRLHF 的 Ray 分布式架构中，Actor（策略网络）和 Reference Model（参考网络）默认部署在不同的 GPU 上。`colocate_actor_ref` 让它们共享同一组 GPU。

**具体节省了什么：**

不开启 colocate 时，需要 4 组 GPU：
- Actor: 2 GPU（ZeRO-3 分片）
- Ref: 2 GPU（ZeRO-3 分片）
- vLLM engine: 1 GPU（推理加速）
- 总计需要 5 GPU，但我们只有 4 张卡

开启 colocate 后：
- Actor + Ref 共享 2 GPU — Ref 模型的权重在 Actor 的 GPU 上加载，不需要额外的 GPU
- vLLM engine: 1 GPU
- 总计只需要 3 GPU（实际用 4 张，多出的 1 张用于 vLLM 或 ZeRO-3 的 offload）

**节省的核心是 Ref 模型的独立 GPU 分配。** Ref 模型只在计算 KL 散度时使用（前向传播，不需要梯度），所以它可以和 Actor 共享 GPU，在 Actor 训练时 Ref 的权重只占显存不占计算。

对于 3B 模型（bf16），Ref 权重约 6GB。colocate 后这 6GB 从"独占 2 张 GPU"变成"分摊到 Actor 的 2 张 GPU 上"，每张卡多占 3GB 但省出了 2 张完整的 GPU。

**不开启 ZeRO-3 时的显存瓶颈：**

3B 模型（bf16）的显存占用估算：
- 模型权重：~6GB
- 优化器状态（AdamW）：~24GB（fp32 参数副本 + 一阶/二阶动量，各 8GB）
- 梯度：~6GB
- 激活值（gradient checkpointing 开启）：~4-8GB（取决于 batch size 和序列长度）
- KV Cache（vLLM rollout）：~2-4GB

单卡总计：~42-48GB，刚好卡在 A6000 的 48GB 边界。

**主要瓶颈是优化器状态（24GB）。** ZeRO-3 将参数、梯度、优化器状态三者都分片到多卡上，2 卡分片后每卡优化器状态降到 12GB，总显存降到 ~28GB，留出足够空间给 rollout 的 KV Cache 和激活值。

### 2.3 训练稳定性：除长度 Hacking 外的 Reward Hacking 现象

**观察到的 Reward Hacking 现象：**

**（1）格式捷径（Format Shortcut）**

这是最显著的 reward hacking。从报告 05 的分析可以看到：v2/v3 训练后，score 的提升完全来自格式合规率（68% → 97%），exec_score 始终为 0。模型学会了"只要输出 `<think>` 和 ` ```python ` 标记就能稳定拿到 0.1 分"，而不去优化代码正确性。

这本质上是因为 format_reward（0.1 分）是确定性的、容易获得的，而 exec_score 在 reward bug 未修复前几乎不可能获得。模型理性地选择了"确定的 0.1 分"而非"几乎不可能的 0.0-1.0 分"。

**检测方式**：对比 format_score 和 exec_score 的提升曲线。如果 format 快速上升而 exec 停滞，就是格式捷径。

**干预方式**：
- 修复 reward bug（报告 06），让 exec_score 能产生非零信号
- 降低 format_reward 的权重（从 0.1 降到 0.05），减少格式捷径的吸引力
- 当前实现中 format_reward 固定为 0.1，未做动态调整

**（2）代码模板化（Template Collapse）**

从 v5 训练日志观察到 group_reward_std 从 0.174 持续下降到 0.160，同时 response_length 从 543 降到 352。这说明模型的输出在趋于一致——对于不同的题目，模型倾向于生成结构相似的代码模板。

这不完全是 hacking，但它降低了 GRPO 的信号质量：当组内所有采样都很相似时，advantage 估计的方差趋近于 0，梯度信号消失。

**检测方式**：监控 group_reward_std 和 response_length 的趋势。如果两者同时持续下降，说明模型在收敛到固定模板。

**干预方式**：
- 提高 temperature（当前 1.0，可以尝试 1.2）增加采样多样性
- 增大 n_samples_per_prompt（8 → 16/64）让组内有更多不同的尝试
- 这也是 stage2 r64 效果更好的原因之一

**（3）KL 漂移导致的"伪提升"**

v4 训练中 KL 从 0.008 飙升到 0.027（3 个 epoch），同时 score 从 0.135 升到 0.167。但独立测试集评测显示 v4 的 exec_score 反而低于 baseline（0.378 vs 0.465）。

这说明 v4 的 score 提升是"伪提升"——模型偏离了预训练分布，在训练集上过拟合，但泛化能力下降。KL 散度过大本身就是一种 reward hacking 的信号：模型找到了一个远离预训练分布但在训练集上得分更高的局部最优。

**检测方式**：监控 KL 散度。如果 KL > 0.02 且仍在快速上升，需要警惕。

**干预方式**：
- v5 将 init_kl_coef 从 0.01 提高到 0.02，限制策略偏移
- 减少 epoch 数（3 → 1），避免重复训练加剧过拟合
- 使用独立测试集评测，而非训练集评测（这正是我们今天做的工作）

## 三、数据与场景理解

### 3.1 课程学习：两阶段数据分布的考量

**Stage 1（2847 题）的组成：**
- Introductory 2397 题（84%）：基础编程题，如数组操作、字符串处理、简单数学
- Interview fn_name 型 450 题（16%）：中等难度，如链表、树、动态规划的简单变体

**Stage 2（360 题）的组成：**
- Competition 360 题（100%）：算法竞赛题，涉及图论、高级 DP、数据结构、数论

**"由易到难"的设计考量：**

1. **Reward 信号密度**：Introductory 题目的通过率远高于 competition（baseline 在 introductory 上 exec=0.465，competition 上接近 0）。Stage 1 用简单题保证模型能获得足够的正向 reward 信号来学习基本的代码生成能力和格式规范。如果一开始就用 competition 题，模型几乎拿不到 exec 分，GRPO 退化为纯格式优化。

2. **能力递进**：先学会写正确的简单代码（函数定义、基本逻辑），再尝试解决需要复杂算法的难题。这类似于人类学编程的过程——先学 for 循环，再学动态规划。

3. **数据量的不对称**：Stage 1 用 2847 题是因为简单题的数据充足且 reward 可靠；Stage 2 只有 360 题是因为 APPS 中 competition 难度的有效数据就这么多（过滤后）。这不是刻意设计的比例，而是数据可用性的约束。

**Stage 2 难题场景下模型最容易出现的逻辑错误：**

从参考解法测试和训练日志分析，主要有三类：

1. **算法选择错误**：模型倾向于用暴力搜索（O(n²) 或 O(n³)）解决需要 O(n log n) 或 O(n) 算法的问题。例如需要用线段树的区间查询问题，模型会写嵌套循环。这在小测试用例上可能通过，但大测试用例会超时。

2. **边界条件处理不当**：competition 题目的测试用例通常包含极端情况（n=1, n=10⁵, 空输入等）。模型生成的代码在"正常"输入上可能正确，但在边界条件上出错。这也是为什么很多样本是 partial pass（部分测试用例通过）而非 full pass。

3. **I/O 格式解析错误**：competition 题目的输入格式通常比较复杂（多行、多组测试用例、嵌套结构）。模型经常在解析输入时出错，比如忘记处理多组测试用例（`for _ in range(T)`），或者把空格分隔的数字当成换行分隔。

### 3.2 可验证奖励（RLVR）的局限性与改进方向

**当前 Reward 设计：**

$$R = \underbrace{\frac{\text{passed\_cases}}{\text{total\_cases}}}_{\text{exec\_score} \in [0,1]} + \underbrace{0.1 \cdot \mathbb{1}[\text{format\_ok}]}_{\text{format\_score}}$$

**单一 Pass Rate 是否足以产出"高质量"代码？**

不够。Pass Rate 只衡量功能正确性（"代码能不能跑对"），不衡量：

1. **时间复杂度**：O(n²) 和 O(n) 的解法只要都能在 2 秒内通过测试用例，得分相同。但在实际场景中，O(n²) 在大数据量下会超时。当前的 2 秒超时在小测试用例上几乎不构成筛选压力。

2. **代码可读性**：变量名 `a, b, c` 和 `left_pointer, right_pointer, result` 得分相同。模型倾向于生成短变量名（因为 overlong_penalty 惩罚长度），这实际上是在鼓励不可读的代码。

3. **代码健壮性**：没有错误处理、没有类型检查的代码只要通过测试用例就满分。

4. **代码风格**：不符合 PEP 8、没有 docstring、magic number 等问题不会被扣分。

**但 Pass Rate 在当前阶段是正确的选择：**

对于 3B 模型，"写出功能正确的代码"本身就是一个很大的挑战（baseline 在 introductory 上只有 46.5% 的 exec_score）。在模型还不能稳定写对代码的阶段，优化时间复杂度或可读性是过早优化。先把正确性做到 80%+，再考虑质量维度。

**如果要优化这些维度，Reward 函数的重新设计：**

```
R = w1 * correctness + w2 * efficiency + w3 * style + format_bonus

其中：
correctness = passed_cases / total_cases                    # 当前已有
efficiency  = 1.0 if time < T/10 else (T - time) / T       # 执行时间越短分越高
style       = pylint_score / 10.0                           # 静态分析评分
format_bonus = 0.05 * has_think_and_code_block              # 降低格式分权重
```

具体改进方向：

1. **分级时间奖励**：不只是"2 秒内通过/超时"的二元判断，而是根据执行时间给梯度奖励。比如 0.1 秒内通过得 1.0，0.5 秒得 0.8，1.5 秒得 0.3。这会自然地鼓励模型选择更高效的算法。实现上只需要修改 `execute_code` 返回执行时间，然后做一个时间到分数的映射。

2. **测试用例加权**：给边界条件和大输入的测试用例更高的权重。当前所有测试用例等权（1/n），但通过大输入测试用例应该比通过小输入更有价值，因为它隐含了算法复杂度的约束。

3. **静态分析奖励**：用 pylint 或 flake8 对生成的代码做静态检查，将评分作为 reward 的一个维度。但这会显著增加评分时间（每次调用 pylint 约 0.5-1 秒），需要权衡。

4. **对比奖励**：不只是和测试用例比，还和参考解法比。如果模型的解法比参考解法更快（执行时间更短），给额外奖励。这需要预先跑一遍参考解法的执行时间作为 baseline。

**实际建议的优先级：**

- 短期（投入产出比最高）：分级时间奖励 + 降低 format_reward 权重
- 中期：测试用例加权 + 修复浮点精度匹配
- 长期：静态分析奖励 + 对比奖励
