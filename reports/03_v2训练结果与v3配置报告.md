## 二、v3 训练配置（已启动，正在运行）

### 2.1 v3 相对 v2 的改动

| 参数 | v2 | v3 | 改动原因 |
|---|---|---|---|
| pretrain | Qwen2.5-Coder-3B-Instruct 原始模型 | **v2 checkpoint** | 从 v2 训练结果继续，不从头开始 |
| max_epochs | 1 | **3** | 1000 条数据跑 1 遍太少，多轮学习提升效果 |
| temperature | 1.0（默认） | **1.2** | 增加采样多样性，防止 group_reward_std 衰减，保持 GRPO 信号 |
| actor_learning_rate | 1e-6（默认） | **5e-7** | 从 checkpoint 继续训练用更小学习率，避免破坏已学到的能力 |
| vllm_gpu_memory_utilization | 0.3 | **0.8** | 提高 GPU 2 利用率，加速 generation 阶段 |
| save_steps | 无 | **20** | 每 20 步保存中间 checkpoint，方便挑选最优模型 |

### 2.2 不变的关键参数

- max_new_tokens = 768
- overlong_buffer_len = 256（惩罚起点 = 512 token）
- overlong_penalty_factor = 1.0
- stop_properly_penalty_coef = 0.0（截断样本奖励归零）
- train_batch_size = 128, rollout_batch_size = 16
- n_samples_per_prompt = 8
- ZeRO Stage 3

### 2.3 完整训练脚本

```bash
python -m openrlhf.cli.train_ppo_ray \
    --ref_num_nodes 1 \
    --ref_num_gpus_per_node 2 \
    --colocate_actor_ref \
    --reward_num_nodes 0 \
    --reward_num_gpus_per_node 0 \
    --critic_num_nodes 0 \
    --critic_num_gpus_per_node 0 \
    --actor_num_nodes 1 \
    --actor_num_gpus_per_node 2 \
    --vllm_num_engines 1 \
    --vllm_tensor_parallel_size 1 \
    --vllm_gpu_memory_utilization 0.8 \
    --pretrain /mnt/sdc/ubuntu/cjz_projects/DRIVE/checkpoints/stage1_v2 \
    --save_path ./checkpoints/stage1_v3 \
    --input_key prompt \
    --label_key test_cases \
    --prompt_data /mnt/sdc/ubuntu/cjz_projects/DRIVE/data/stage1_base_str.jsonl \
    --input_template "{}" \
    --max_samples 1000 \
    --micro_train_batch_size 1 \
    --train_batch_size 128 \
    --rollout_batch_size 16 \
    --n_samples_per_prompt 8 \
    --max_new_tokens 768 \
    --max_epochs 3 \
    --zero_stage 3 \
    --gradient_checkpointing \
    --param_dtype bf16 \
    --advantage_estimator group_norm \
    --use_kl_loss \
    --init_kl_coef 0.01 \
    --temperature 1.2 \
    --actor_learning_rate 5e-7 \
    --remote_rm_url http://localhost:5000/scores \
    --stop_properly_penalty_coef 0.0 \
    --overlong_buffer_len 256 \
    --overlong_penalty_factor 1.0 \
    --use_tensorboard ./checkpoints/stage1_v3 \
    --save_steps 20
```

## 三、今天踩过的坑

### 3.1 max_new_tokens 512 导致截断率飙升

- 曾尝试将 max_new_tokens 从 768 降到 512 以鼓励短代码
- 同时 overlong_buffer_len=128，惩罚起点 = 384 token
- 结果：截断率从 11.7% 飙到 35.9%，reward 变负（-0.088）
- 原因：v2 模型习惯生成 ~364 token（均值），但分布有长尾，很多回复在 384-768 之间
- 结论：不应通过硬截断限制长度，应靠渐进惩罚引导。已改回 768 + buffer=256

### 3.2 rollout_batch_size=64 反而拖慢训练

- 曾尝试将 rollout_batch_size 从 16 提到 64 以提高 GPU 利用率
- 结果：每步从 156s 暴增到 1077s（6.9x），其中 ppo_train 从 77s 到 835s（10.8x）
- 原因：ZeRO-3 通信开销随 batch 超线性增长
- 结论：在 ZeRO-3 下，小 rollout_batch_size 反而更快。已改回 16

### 3.3 TensorBoard 启动报错 pkg_resources

- `ModuleNotFoundError: No module named 'pkg_resources'`
- 解决：`pip install --force-reinstall setuptools`
- 但 `tensorboard` 命令仍有问题，改用 `python -m tensorboard.main` 启动成功
- 注意：vllm 0.19.0 和 transformers 5.5.3 有版本冲突警告，但不影响训练

### 3.4 ZeRO-2 不可用

- GPU 0/1 显存已用到 90%+，ZeRO-2 需要聚合完整梯度分区会 OOM
- 必须保持 ZeRO-3

## 四、GPU 分配方案（4× RTX A6000）

| GPU | 角色 | 典型显存占用 | 说明 |
|---|---|---|---|
| 0 | Actor shard + Ref shard + python workers | ~46 GB (94%) | 主力计算卡 |
| 1 | Actor shard + Ref shard | ~44 GB (90%) | 主力计算卡 |
| 2 | vLLM 引擎 | ~39 GB (0.8 利用率) | generation 阶段活跃，ppo_train 阶段空闲 |
| 3 | Ray 调度 / PPOTrainer 主进程 | ~18 MB | 几乎不用显存，是 4 卡 Ray 架构的固有浪费 |

GPU 2 和 3 利用率低是 Ray 架构的正常现象：vLLM 和 PPO 训练交替执行，不能同时跑。

## 五、明天的工作

### 5.1 检查 v3 训练结果

- 查看 `./checkpoints/stage1_v3/` 下的 checkpoint 和 TensorBoard 日志
- 重点关注指标：
  - score 是否持续上升（目标 > 0.15）
  - truncated_rate 是否保持 < 20%
  - group_reward_std 是否保持 > 0.03（GRPO 信号有效）
  - kl 是否在合理范围（< 0.02）
- 对比 3 个 epoch 的学习曲线，看是否有过拟合迹象（score 先升后降）

### 5.2 挑选最优 checkpoint

- save_steps=20，会保存多个中间 checkpoint
- 选 score 最高且 truncated_rate 最低的那个作为最终模型

### 5.3 后续优化方向

- 如果 3 个 epoch 效果好：可以考虑增加数据量（当前只有 1000 条）
- 如果 score 提升有限：考虑调整 n_samples_per_prompt（8→16）增加 GRPO 组内对比
- 如果过拟合明显：减少 epoch 或增大 KL 系数
- Stage 2 训练：使用更难的编程题数据集进行进阶训练
