## 四、Stage2 数据修复

### 4.1 问题

Stage2 数据（competition 难度）中有 5 个样本的 input 是 list 而非 string，这是 APPS 数据集本身的格式不一致。在修复后的 `_is_stdin_type()` 逻辑下，这 5 个样本会被错误分类为 function-call 型（没有 fn_name 且 input 不是 str）。

### 4.2 修复

在 `prepare_data.py` 的 `get_test_cases()` 中增加归一化逻辑：

```python
if not fn_name:
    if isinstance(inp, list):
        inp = "\n".join(str(x) for x in inp)
    if isinstance(out, list):
        out = "\n".join(str(x) for x in out)
```

此修改仅在 `not fn_name`（stdin 型）条件下触发，不影响 stage1 数据。

## 五、Stage2 消融实验设计

### 5.1 实验目标

研究 GRPO 组大小（n_samples_per_prompt）对 competition 难题训练效果的影响。

### 5.2 实验设计

| | Rollout=16（对照组） | Rollout=64（实验组） |
|---|---|---|
| 脚本 | `train_stage2_r16.sh` | `train_stage2_r64.sh` |
| n_samples_per_prompt | 16 | 64 |
| rollout_batch_size | 8 | 2 |
| 每步总样本数 | 128 | 128 |
| 每步题目数 | 8 道 | 2 道 |
| 每题回答数 | 16 个 | 64 个 |
| **总步数** | **90** | **90** |
| 总生成量 | 11,520 | 11,520 |
| 题目覆盖 | 全部 360 题 × 2 遍 | 随机 180 题 × 1 遍 |
| 控制方式 | max_epochs=2, max_samples=360 | max_epochs=1, max_samples=180 |
| 预计时间 | ~4 小时 | ~4 小时 |

### 5.3 控制变量

两组实验共享以下参数：
- pretrain：`checkpoints/stage1_v5_step135`
- actor_learning_rate：3e-7
- init_kl_coef：0.03
- train_batch_size：128
- temperature：1.0
- max_new_tokens：768

### 5.4 核心假设

- **r64 优势**：每题 64 个回答，GRPO 组内对比信号更强。对于 competition 难题（大部分回答得 0 分），64 个样本中更可能出现至少 1 个正确回答，从而产生有效的正向梯度信号。
- **r16 优势**：每步看 8 道不同的题，题目多样性更高，2 epoch 覆盖全部 360 题两遍。r64 只看 180 题一遍，可能遗漏部分题型。
- **预期**：如果 competition 题的通过率极低（<5%），r64 应该更有优势；如果通过率尚可（>10%），r16 的多样性优势可能更重要。

### 5.5 评估方案

两组训练完成后，使用相同的评估集（`data/stage2_v2.jsonl` 全部 360 题）进行离线评估，对比：
1. 总分（score）
2. 执行通过率（exec_score > 0 的比例）
3. 完全通过数（exec_score = 1.0）
4. 格式合规率
