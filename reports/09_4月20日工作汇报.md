## 三、发现并修复的严重 Bug：fn_name 字段丢失

### 3.1 问题描述

APPS 数据集有两种 test case 格式：
- **fn_name 型**（函数调用型）：`input` 是参数列表（如 `['AAAA']`），`output` 是返回值，附带 `fn_name` 字段
- **stdin 型**：`input` 是纯字符串（如 `'6\n1\n2\n'`），`output` 是字符串

`prepare_data.py` 在生成训练数据时**丢弃了 `fn_name` 字段**，导致 `reward_service.py` 无法正确区分两种类型。

### 3.2 影响范围

`reward_service.py` 的 `_is_stdin_type()` 原始逻辑：
```python
def _is_stdin_type(tc):
    inp = tc["input"]
    return isinstance(inp, list) and len(inp) == 1 and isinstance(inp[0], str)
```

这个逻辑将**单参数的 fn_name 型样本**（如 `input: ['AAAA']`）错误地分类为 stdin 型。

在 1000 条训练数据中：
- 715 条正确识别为 function-call 型（多参数）
- 55 条正确识别为 stdin 型
- **230 条被错误分类**（23%）：fn_name 型但被当作 stdin 型处理

这 230 条数据在训练过程中**永远无法获得正确的 exec_score**，给模型提供了错误的负向信号。

### 3.3 修复方案

**`prepare_data.py`**：`get_test_cases()` 保留 `fn_name` 字段
```python
def get_test_cases(r):
    io = json.loads(r["input_output"])
    ins, outs = io["inputs"], io["outputs"]
    fn_name = io.get("fn_name", None)
    cases = []
    for i in range(min(5, len(ins))):
        tc = {"input": ins[i], "output": outs[i]}
        if fn_name:
            tc["fn_name"] = fn_name
        cases.append(tc)
    return cases
```

**`reward_service.py`**：`_is_stdin_type()` 优先检查 `fn_name` 字段
```python
def _is_stdin_type(tc):
    if "fn_name" in tc:
        return False
    return isinstance(tc["input"], str)
```

同时修复了 `_execute_func()` 中 output 为 list 时的解包问题，以及 `_compute()` 中 stdin 分支对纯字符串 input 的适配。

### 3.4 验证结果

修复后用 DNA_strand 样本（单参数 fn_name 型，之前会被误分类）测试，正确得分 1.1（满分）。新数据 2847 条，0 条误分类。

### 4.2 新数据组成

Stage1 v2 数据：**全部 introductory（2397 条）+ interview fn_name 型（450 条）= 2847 条**

选择理由：
- fn_name 型的 reward 评分最可靠（直接调用函数比较返回值）
- interview fn_name 型和 introductory 风格一致（都是写函数），难度适中
- interview stdin 型难度偏高且 reward 评分对 stdin 型不够稳定，暂不纳入

### 4.3 数据文件说明

生成了两个版本，原始数据完整保留：
- `data/stage1_v2.jsonl`：`test_cases` 为原生 JSON list，用于离线评测和调试
- `data/stage1_v2_str.jsonl`：`test_cases` 序列化为 JSON 字符串，用于 OpenRLHF 训练（OpenRLHF 要求 label 字段为字符串类型）

## 六、后续优化建议

### 6.1 短期（明天可做）

1. **v5 训练完成后跑离线评估**，与 v4 和 baseline 三方对比
2. **挑选最优 checkpoint**：v5 保存了 step 15/30/45/.../的 checkpoint，可以分别评估找最优点
3. **分析失败样本**：重点看 v5 新增通过的题目和仍然失败的题目，找规律

### 6.2 中期优化方向

1. **增加 n_samples_per_prompt**（8→16）：增强 GRPO 组内对比信号，代价是 rollout 时间翻倍但 generation 只占总时间 17%
2. **调整 reward 结构**：
   - 格式分从 0.1 降到 0.05，减少格式捷径的吸引力
   - 引入分级 exec reward：代码能运行不报错 +0.1，部分 test case 通过按比例给分
3. **多 epoch 训练**：如果 v5 单 epoch 效果好，可以尝试 2 epoch（降低 lr 到 3e-7）
4. **Stage2 训练**：用 v5 最优 checkpoint 作为基础，在 competition 难度数据上继续训练

### 6.3 长期方向

1. **扩充外部数据**：从 HumanEval、MBPP、CodeContests 等公开数据集补充
2. **改进 reward 对 stdin 型的支持**：当前 stdin 型评分不够稳定，限制了可用数据范围
3. **显存优化**：探索 offload 或 LoRA 方案，支持更长的 max_new_tokens
